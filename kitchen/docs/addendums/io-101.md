# Io 101

If you are going to use `std.Io` directly - this page for you.

(BTW - API Reference and Patterns pages assume your Io knowledge)


---

## The four things `std.Io` gives you

```text
Io              access to the runtime вЂ” passed to anything that waits or spawns
Future(T)       a result that isn't ready yet
Io.Select       waits on several event sources at once
Io.Group        runs several tasks, waits for all of them
```

- `Io` вЂ” think of it as a capability token. Anything that needs threads, timers, or
  waiting takes one.

- `Future(T)` вЂ” you get the value by calling `.await()`.
- `Io.Select` вЂ” internally: `queue: Queue(U)` + `group: Group`. `select.await()` reads
  the next completed result from the queue.

- `Io.Group` вЂ” runs several tasks concurrently, waits for all of them, or cancels all
  of them.

## Future вЂ” one asynchronous result

```zig
var fut = try io.concurrent(workerFn, .{&ctx});
// ... do other work ...
try fut.await(io);   // blocks until worker exits; returns worker's return type
```

```text
  concurrent() в”Ђв”Ђв–є Future(T)
                      в”‚
              в”Њв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”јв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”ђ
              в–ј               в–ј
          .await()        .cancel(io)
              в”‚               в”‚
              в–ј               в–ј
           result      error.Canceled
```

Rules:

- `args` is a tuple вЂ” `.{arg1, arg2, ...}` вЂ” passed verbatim to the function.
- `io.concurrent` copies `args` before returning. Stack-allocated args are safe вЂ” no
  heap context needed.

- No `io` is injected automatically. If the worker needs it, pass it explicitly:
  `.{io, &ctx}`.

- `fut.await(io)` returns the worker's return type directly.
- `fut.cancel(io)` injects `error.Canceled` at the worker's next cancellation point,
  then awaits.

- `Future` is a resource вЂ” call `await` or `cancel` exactly once.

## Io.Group вЂ” several tasks, one wait

```zig
var group: std.Io.Group = .init;
defer group.cancel(io);   // safe: no-op if already awaited

try group.concurrent(io, workerFn, .{&ctx1});
try group.concurrent(io, workerFn, .{&ctx2});
try group.concurrent(io, workerFn, .{&ctx3});

try group.await(io);   // blocks until all workers exit
```

Rules:

- Worker return type must be coercible to `Cancelable!void` вЂ” `void`, `!void`, and
  `Cancelable!void` all work.

- `error.Canceled` returned by a worker is swallowed at the Group boundary вЂ” it's a
  cancellation propagation stop, not an error you see.

- `group.await(io)` returns `Cancelable!void`.
- `group.cancel(io)` injects `error.Canceled` into every running worker, then waits.
  Safe to call again after an `await` вЂ” a no-op.

- `group.concurrent` after `group.await` starts a new round in the same Group.

## Io.Select вЂ” several event sources, one loop

```zig
var buf: [8]MasterEvent = undefined;
var sel: std.Io.Select(MasterEvent) = std.Io.Select(MasterEvent).init(io, &buf);

try sel.concurrent(.inbox, mailbox.receiveResult, .{ mbh, null });
try sel.concurrent(.pool_ev, pool.getWaitResult, .{ ph, TAG, null });

while (true) {
    const event = try sel.await();
    switch (event) {
        .inbox => |r| { /* process, then re-register */ },
        .pool_ev => |r| { /* process, then re-register */ },
    }
}
```

The rhythm:

```text
register
    в†“
await
    в†“
process
    в†“
register again
```

Internals (verified from `std/Io.zig`):

- `queue: Queue(U)` вЂ” completed results land here.
- `group: Group` вЂ” owns every spawned concurrent task.
- `select.await()` is `queue.getOne(io)` вЂ” blocks until the next result arrives.
- `select.concurrent(field, fn, args)` spawns `fn` via `group`, wraps its return value
  as `@unionInit(U, @tagName(field), result)`, and pushes it onto `queue`.

Direct push вЂ” no spawn, for a result that's already available or an external callback:

```zig
select.queue.putOneUncancelable(select.io, .{ .field = value }) catch {};
```

## Event sources вЂ” the pattern behind Select

An event source is a blocking function passed to `select.concurrent`. When it returns,  
the result lands in the queue.

```text
  blockingFn в”Ђв”Ђв”Ђв”Ђв–є select.concurrent(field, blockingFn, args)
                        в”‚
                        в–ј (fn runs, result в†’ queue)
                   Io.Select.queue
                      в”‚   в”‚
                      в–ј   в–ј
               completed  canceled
               (result)   (error.Canceled)
```

Two ways to produce an event:

1. `select.concurrent(field, blockingFn, args)` вЂ” spawn a blocking function.
2. `select.queue.putOneUncancelable(io, value)` вЂ” push directly, no spawn.

Matryoshka's Mailbox and Pool plug into exactly this pattern вЂ” see  
[API Reference вЂ” Mailbox event source helpers](../api/mailbox.md) and  
[API Reference вЂ” Pool event source helpers](../api/pool.md).

## Cancellation

A function that waits вЂ” for data, for a timeout, for a condition вЂ” can be canceled by  
the runtime.

- If a function can be canceled, its return type includes `Cancelable` in the error
  union. The signature is the only source of truth for this.

- Cancel is something you do *to* a Future or a Group вЂ” it never happens on its own.

```text
  concurrent() в”Ђв”Ђв–є Future(T)

              в–ј               в–ј
          .await()        .cancel(io)
              в”‚               в”‚
              в–ј               в–ј

```
